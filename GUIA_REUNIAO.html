Git — Controle de Versão

Como o deploy funciona? Vocês usam CI/CD? Comum
Deploy manual por enquanto. O processo é: fazer o push para a branch main no GitHub, entrar no servidor via SSH, buscar as mudanças com git fetch origin e aplicar com git reset --hard origin/main, depois reiniciar o serviço com systemctl restart agente-comexim.
💡 Se perguntarem "por que não git pull?": pull pode falhar silenciosamente se o branch local tiver divergido. O reset --hard garante que o servidor fica idêntico ao repositório.
Qual é a estratégia de branching? Provável
Fluxo simples: tudo vai para main. Develop direto na main, deploy direto da main. Para um time pequeno isso funciona bem e evita complexidade desnecessária. Se o time quiser adotar Gitflow (com branch develop e feature branches), é só uma questão de convenção — a estrutura do projeto suporta.
O .env está no repositório? Importante
Não. O .env está no .gitignore — nunca vai pro repositório. O que existe no repo é um .env.example com todas as variáveis listadas mas sem valores reais. No servidor, o .env com as credenciais reais fica só na máquina.
Tem algum hook ou pipeline configurado? Técnico
Não por enquanto. Não tem GitHub Actions, não tem webhook de deploy automático. O deploy é manual via SSH. Isso pode ser melhorado futuramente com um GitHub Action que faz SSH no servidor e executa os comandos de deploy automaticamente.

Arquitetura do Sistema

Por que FastAPI e não Django ou Flask? Técnico
Performance e async nativo. O FastAPI é assíncrono por padrão (baseado em Starlette + asyncio), o que é essencial aqui porque precisamos processar múltiplas mensagens simultâneas sem bloquear. Flask é síncrono. Django tem muito overhead desnecessário para uma API de webhook.
O que é LangGraph? Por que não LangChain direto? IA
LangGraph é uma camada acima do LangChain que permite criar agentes com estado e fluxos complexos de decisão. O agente usa o padrão ReAct (Reason + Act): a IA raciocina sobre qual ferramenta usar, executa, analisa o resultado e decide se precisa de mais ferramentas. LangChain puro não tem controle de fluxo tão estruturado.
Como funciona a integração com o WhatsApp? Comum
Via UAZAPI, que é um intermediário que conecta ao WhatsApp Web. O fluxo é: usuário manda mensagem → UAZAPI recebe → faz um POST para o nosso servidor (/webhook) → nossa API processa → chama a IA → manda a resposta de volta via UAZAPI. A gente não acessa o WhatsApp diretamente — o UAZAPI faz essa parte.
Por que Redis além do Supabase? Não dá pra usar só um? Técnico
Papéis diferentes. Redis é in-memory — extremamente rápido para coisas que mudam a todo momento: histórico de conversa, buffer anti-flood, cache de autenticação. Supabase é PostgreSQL persistente — usado para agendamentos e preferências que precisam sobreviver a restart do servidor. Misturar os dois daria lentidão no Redis ou perda de dados no Supabase.
O sistema escala? Dá pra rodar em múltiplas instâncias? Avançado
Sim, com ressalvas. Redis e Supabase são externos — qualquer instância acessa os mesmos dados. O gargalo seria o SQL Server, que já tem sua própria gestão de conexões. Para escalar horizontalmente, precisaria de um load balancer na frente e garantir que o scheduler (APScheduler) rode em apenas uma instância para não duplicar envios.

Deploy e Servidor

O que é systemd? Por que usar ele? Linux
Systemd é o gerenciador de processos do Linux. Garante que o serviço sobe automaticamente quando o servidor reinicia, reinicia sozinho se cair, e centraliza os logs (acessíveis via journalctl). A alternativa seria usar screen/tmux ou PM2 — systemd é a forma mais robusta e padrão em produção Linux.
Como ver se o serviço está rodando? Operação
systemctl status agente-comexim — mostra se está ativo, há quanto tempo, e as últimas linhas de log. Para ver logs completos em tempo real: journalctl -u agente-comexim -f.
⏱️ Lembra que o servidor registra em UTC. Brasília é UTC-3. Mensagem às 14h SP = 17h nos logs.
Como funciona o ambiente virtual Python (venv)? Python
Isola as dependências do projeto do Python do sistema. O requirements.txt lista todas as bibliotecas necessárias. No servidor, o venv fica em /opt/agente-comexim-whatsapp/venv/. Se instalar um pacote novo, precisa ativar o venv e rodar pip install -r requirements.txt antes de reiniciar o serviço.
Tem Docker? Por que não containerizou? Infra
Não tem Docker nesta versão. Roda direto no servidor Linux com Python + venv + systemd. Containerizar seria a próxima evolução natural — facilitaria o setup em novos servidores e o ambiente de teste. O ODBC Driver do SQL Server dentro de container tem algumas particularidades de instalação que complicam um pouco, mas é totalmente factível.

Banco de Dados

O sistema escreve alguma coisa no Protheus? Crítico
Não. Zero escrita. Todas as consultas são SELECT * FROM stored_function(). O sistema é 100% read-only no Protheus. Isso foi uma decisão de segurança intencional — a IA nunca modifica dados do ERP.
Por que stored functions e não queries SQL diretas? SQL
Segurança e controle. As stored functions encapsulam a lógica de negócio no banco — o sistema só chama SELECT * FROM dbo.IA_Vendas() e recebe os dados formatados. Não tem SQL dinâmico perigoso, não tem acesso direto às tabelas do Protheus. Qualquer mudança na estrutura de dados fica isolada dentro da function, sem precisar mexer no código Python.
O que é ODBC e por que é necessário? Técnico
ODBC (Open Database Connectivity) é um driver de conexão padronizado. O Python usa a biblioteca pyodbc para se conectar ao SQL Server, e o pyodbc precisa do driver ODBC da Microsoft instalado no sistema operacional. É como um "tradutor" entre Python e SQL Server. No Linux, instala com apt-get install msodbcsql17.
Tem ORM? Usa SQLAlchemy? Técnico
Não tem ORM. Acesso direto via pyodbc. Como o sistema só chama stored functions (não monta queries complexas nem mapeia entidades), um ORM adicionaria complexidade sem benefício. O arquivo app/core/database.py tem a classe SQLServerClient que gerencia toda a conexão.

Inteligência Artificial

Como a IA sabe qual consulta fazer? IA
O LLM (GPT-4o) decide. Cada "tool" tem uma descrição em linguagem natural. A IA lê a pergunta do usuário, compara com as descrições das tools disponíveis, e escolhe qual chamar. Por exemplo, se o usuário perguntar "quanto foi vendido em janeiro?", a IA escolhe a tool pesquisa_vendas automaticamente.
O histórico de conversa fica salvo para sempre? Privacidade
Não. O histórico fica no Redis com TTL de 2 horas (REDIS_MEMORY_TTL=7200). Depois de 2 horas de inatividade, expira automaticamente. Não tem banco de dados persistente de mensagens — por design, para não guardar dados sensíveis.
E se a OpenAI ficar fora do ar? O sistema cai? Risco
Sim, o sistema fica sem resposta. A dependência da OpenAI é crítica — sem ela, a IA não funciona. O sistema retorna uma mensagem de erro amigável ao usuário, mas não tem fallback para outro modelo. Isso é uma limitação conhecida que pode ser endereçada futuramente com suporte a múltiplos provedores (ex: Azure OpenAI como backup).
O que é rate limit e como está tratado? Operação
Rate limit é o limite de requisições por minuto na API da OpenAI. O plano básico tem 30.000 tokens por minuto (TPM). Se várias pessoas usarem ao mesmo tempo com contexto grande, pode estourar. O sistema faz retry automático — espera 10 segundos e tenta de novo. Se o crédito da conta zerar (insufficient_quota), aí não tem retry — precisa adicionar crédito manualmente.

Perguntas Difíceis — Como Responder

Por que não tem testes automatizados? Difícil
Honest answer: O projeto foi entregue em MVP (versão inicial funcionando). Testes automatizados são o próximo passo natural. A dificuldade principal é que a maior parte da lógica depende de serviços externos (OpenAI, SQL Server, Redis) — os testes precisariam de mocks bem estruturados. É tecnicamente viável e recomendado adicionar cobertura de testes na evolução do projeto.
Tem documentação de API? Swagger? Técnico
Sim, automaticamente. FastAPI gera Swagger UI automático. Basta acessar http://servidor:8000/docs enquanto o serviço está rodando. Mas a única rota relevante externamente é o POST /webhook — as demais são internas.
Como adicionamos uma nova consulta (nova stored function)? Evolução
4 passos: 1) Criar a stored function no SQL Server. 2) Registrar em FUNCTION_PARAMETERS no database.py. 3) Criar a tool em sql_tools.py com descrição em linguagem natural. 4) Adicionar na lista get_tools(). Sem mais nada — a IA aprende automaticamente a usar a nova tool pela descrição.
Tem logs estruturados? Sentry? Datadog? Observabilidade
Logs via Python logging + journalctl por enquanto. Não tem Sentry nem Datadog nesta versão. Os logs já são bastante detalhados (query executada, tempo, erros com stack trace). A próxima evolução seria integrar um serviço de monitoramento de erros como Sentry — é uma adição pequena (instalar o SDK e adicionar uma linha no código).

Glossário Rápido — Termos que Podem Aparecer

webhook
URL que recebe notificações automáticas de outro sistema. O UAZAPI faz POST no nosso /webhook quando chega mensagem.
endpoint
Uma rota/URL da API. O sistema tem o endpoint POST /webhook como entrada principal.
async / await
Programação assíncrona — permite processar múltiplas coisas ao mesmo tempo sem bloquear o servidor.
TTL (Time to Live)
Tempo de vida de um dado em cache. TTL=7200 significa que expira em 2 horas.
LLM
Large Language Model — o modelo de IA. Neste projeto: GPT-4o da OpenAI.
tool / ferramenta
Função que a IA pode chamar. Cada consulta SQL é uma "tool". A IA escolhe qual usar.
ReAct pattern
Padrão de agente IA: Reason (raciocina) + Act (executa tool) + Observe (analisa resultado) em loop.
systemd service
Processo Linux gerenciado pelo sistema. Sobe automaticamente no boot, reinicia se cair.
venv
Ambiente virtual Python — isola as dependências do projeto para não conflitar com outras aplicações do servidor.
ODBC
Driver de conexão ao SQL Server. Precisa estar instalado no servidor Linux para o Python conectar ao Protheus.
stored function
Função salva no banco SQL Server. O sistema faz SELECT * FROM dbo.FunctionName() para buscar dados.
rate limit
Limite de requisições por minuto na API da OpenAI. Se estourar, o sistema aguarda 10s e tenta novamente.
git reset --hard
Força o código local a ficar idêntico ao repositório remoto. Usado no deploy para garantir atualização correta.
APScheduler
Biblioteca Python de agendamento. Roda jobs em horários definidos — usado para enviar relatórios automáticos.
MVP
Minimum Viable Product — versão inicial funcionando com as funcionalidades essenciais. Base para evoluir.
SSH
Protocolo para acessar o servidor remotamente pelo terminal com segurança.
