Fluxo Completo de uma Mensagem

1

Webhook recebe a mensagem app/api/webhook.py

UAZAPI faz POST para /webhook. O webhook parseia o payload, extrai número, texto e tipo (texto ou áudio). Ignora mensagens enviadas pelo próprio bot (fromMe: true). Agenda process_message_flow() como background task e retorna 200 imediatamente ao UAZAPI.

2

Anti-flood app/core/redis_client.py

Mensagem entra no buffer Redis (buffer:{telefone}). Sistema aguarda ANTI_FLOOD_WAIT_SECONDS (padrão: 20s). Se novas mensagens chegarem, o timer reinicia. Quando o usuário para de digitar, todas as mensagens são concatenadas e processadas juntas.

3

Autenticação app/services/auth.py

Consulta cache Redis (user:{telefone}). Se não estiver em cache, chama POST {PROTHEUS_API_URL}/iaProtheus/getToken. Retorna nome, email e lista de permissões. Fica em cache por REDIS_MEMORY_TTL (2h). Se não autorizado, responde e para.

4

Transcrição de áudio app/services/audio.py — só se for áudio

Baixa o arquivo via UAZAPI, envia para OpenAI Whisper, retorna texto transcrito que segue o fluxo normal.

5

Orquestração app/agents/orchestrator.py

Carrega histórico do Redis (últimas REDIS_MEMORY_WINDOW × 2 mensagens). Detecta fluxo ADA ou consulta normal. Monta system prompt com dados do usuário + data atual. Cria agente LangGraph (ReAct) com as tools disponíveis. Chama OpenAI com todo o contexto.

6

Execução das tools app/agents/sql_tools.py

A IA decide qual tool chamar. A tool executa sql_client.execute_function(nome, filtros) que monta e roda SELECT * FROM dbo.NomeFuncao(params). Resultado volta como lista de dicionários. A IA interpreta e gera resposta em linguagem natural.

7

Formatação e envio formatter.py + whatsapp.py

Se ENABLE_RESPONSE_FORMATTER=true: divide resposta longa em múltiplas mensagens. Envia cada parte com delay de MESSAGE_DELAY_SECONDS. Chama POST {EVOLUTION_API_URL}/send/text do UAZAPI.

8

Memória

Mensagem do usuário e resposta da IA são salvas no Redis. Chave: memory:{session_id} com TTL de REDIS_MEMORY_TTL (2h).

SMTP_PASSWORD para Gmail é uma senha de aplicativo, não a senha da conta. Gerar em: Conta Google → Segurança → Senhas de app.

⚠️

REDIS_MEMORY_WINDOW=10 significa últimas 10 perguntas + 10 respostas em contexto. Aumentar demais estoura o limite de tokens da OpenAI (30k TPM no plano básico).

Deploy em Produção (Linux/systemd)

O serviço está configurado em /etc/systemd/system/agente-comexim.service.

Comandos essenciais

# Ver status
systemctl status agente-comexim

# Reiniciar após atualização
systemctl restart agente-comexim

# Ver logs em tempo real
journalctl -u agente-comexim -f

# Ver últimas 50 linhas
journalctl -u agente-comexim -n 50 --no-pager

# Logs de um período específico (servidor em UTC, Brasília = UTC-3)
journalctl -u agente-comexim --since "2026-05-29 17:25" --until "2026-05-29 17:35" --no-pager

Processo de deploy (passo a passo)

cd /opt/agente-comexim-whatsapp

# 1. Atualizar código — SEMPRE assim, nunca só git pull
git fetch origin
git reset --hard origin/main

# 2. Reiniciar serviço
systemctl restart agente-comexim

# 3. Confirmar que subiu sem erros
journalctl -u agente-comexim -n 20 --no-pager
🚫

Nunca usar só git pull para deploy. Pode dizer "Already up to date" mesmo depois de buscar novos commits se o branch local estiver em estado divergente. Sempre usar git fetch origin && git reset --hard origin/main.

Banco de Dados — SQL Server (Protheus)

✅

O sistema nunca escreve no banco. Todas as consultas são SELECT * FROM stored_function(). Nenhum INSERT, UPDATE ou DELETE.

Como as queries funcionam

O arquivo app/core/database.py tem a classe SQLServerClient com o método execute_function():

# Funções sem parâmetros — filtros viram WHERE
sql_client.execute_function("dbo.IA_Vendas", filters={"mesEmbarque": "202601"})
# → SELECT * FROM dbo.IA_Vendas() WHERE mesEmbarque = '202601'

# Funções Par — parâmetros vão na chamada SQL
sql_client.execute_function("dbo.IA_VendasPar", filters={"data_inicio": "20260101", "data_fim": "20260131"})
# → SELECT * FROM dbo.IA_VendasPar('20260101', '20260131')

Mapeamento de funções (FUNCTION_PARAMETERS em database.py)

Função SQLParâmetrosTipo de filtro
IA_VendasnenhumWHERE (mesEmbarque)
IA_VendasPardata_inicio, data_fimna chamada SQL
IA_ComprasnenhumWHERE (emissao)
IA_ComprasPardata_inicio, data_fimna chamada SQL
IA_ContasAPagarnenhumWHERE (vencimento)
IA_ContasAPagarPardata_inicio, data_fimna chamada SQL
IA_ContasPagasnenhumWHERE (emissao)
IA_ContasPagasPardata_inicio, data_fimna chamada SQL
IA_ContasARecebernenhumWHERE (vencimentoReal)
IA_ContasAReceberPardata_inicio, data_fimna chamada SQL
IA_OrcamentonenhumWHERE (ano, mes)
IA_OrcamentoPardata_inicio, data_fimna chamada SQL (YYYYMM)

Adicionar uma nova função

  1. Criar a stored function no SQL Server
  2. Adicionar entrada em FUNCTION_PARAMETERS em app/core/database.py
  3. Criar a tool correspondente em app/agents/sql_tools.py
  4. Adicionar a tool na lista get_tools() de sql_tools.py

Autenticação de Usuários

Fluxo normal

  1. Usuário manda mensagem pelo WhatsApp
  2. Sistema extrai o telefone (sem prefixo 55)
  3. Consulta POST {PROTHEUS_API_URL}/iaProtheus/getToken com {"telefone": "13991234567"}
  4. Protheus retorna dados do usuário e lista de permissões
  5. Resultado fica em cache no Redis por 2h

Bypass (hardcoded em auth.py)

Alguns usuários estão configurados diretamente em app/services/auth.py para não depender da API do Protheus. Útil para testes e para usuários que a API não retorna corretamente.

Agente IA — Como Funciona

Pattern ReAct (Reason + Act)

📩 Mensagem do usuário
→
🧠 Raciocina sobre qual tool usar
→
⚙️ Executa a tool
→
📊 Analisa resultado
→
🔁 Precisa mais tools?
→
✅ Resposta final

System prompt

Gerado dinamicamente em app/prompts/system_prompt.py com: nome e permissões do usuário, data e hora atual, instruções de comportamento, e lista de tools disponíveis e quando usar cada uma.

Histórico de conversa

O histórico fica no Redis e é carregado a cada mensagem. Limite configurável por REDIS_MEMORY_WINDOW. Janela deslizante — só as últimas N trocas são enviadas para a OpenAI.

Rate limit OpenAI

O sistema tem retry automático em caso de erro 429 (rate limit): aguarda 10 segundos e tenta novamente. Se falhar novamente, retorna mensagem de erro ao usuário.

🚫

Se o erro for insufficient_quota (crédito zerado), não há retry — precisa adicionar crédito na conta OpenAI em platform.openai.com/billing.

Como a tool pesquisa_cotacao funciona

Chama SELECT * FROM dbo.IA_Cotacao_Par('YYYYMMDD'). Se o usuário não informar data, usa o dia atual.

  • data: data desejada (qualquer formato, converte automaticamente para YYYYMMDD)
  • tipo: filtro por tipo ("Dolar", "Euro", "Café", "Dólar Comercial")
  • ativo: filtro por instrumento ("DOLZ26", "KCK6", etc.)

Campos disponíveis no retorno: COTACAO, AJUSTE, ABERTURA, FECHAMENTO, SITUACAO, tipo, ativo.

Scheduler de Relatórios Agendados

Como funciona

O APScheduler roda um job toda hora no minuto 0 que:

  1. Busca no Supabase todos os relatórios com next_run <= agora e status = 'ativo'
  2. Para cada relatório, autentica o usuário e executa a consulta via agente IA
  3. Envia o resultado pelo canal configurado (whatsapp, email ou ambos)
  4. Se frequencia = 'unico': cancela o agendamento após enviar
  5. Se recorrente: atualiza next_run para a próxima execução

Frequências suportadas

FrequênciaDescrição
diarioTodos os dias no horário definido
semanalUma vez por semana no dia/horário definido
mensalUma vez por mês no dia/horário definido
unicoUma única vez no horário definido, depois cancela automaticamente

Email com anexo xlsx

Quando o canal inclui email: salva resultado bruto no Redis (scheduler_result:{telefone}), lê do Redis e gera .xlsx com openpyxl, envia como anexo no email.

O que NÃO Fazer — Armadilhas Conhecidas

🚫

Git no servidor — nunca só git pull

Pode dizer "Already up to date" mesmo com novos commits. Sempre usar:
git fetch origin && git reset --hard origin/main

🚫

Redis e event loop — não trocar o padrão ThreadPoolExecutor

O _salvar_resultado_scheduler usa ThreadPoolExecutor para rodar coroutines de dentro de código síncrono (tools LangChain). Não tentar trocar por asyncio.ensure_future() ou loop.run_until_complete() — vai dar "event loop já rodando".

⚠️

REDIS_MEMORY_WINDOW — não aumentar além de 15

Com GPT-4o e histórico longo (especialmente com resultados de queries grandes), é fácil estourar os 30.000 TPM do plano básico da OpenAI.

⚠️

ODBC Driver — primeiro passo que esquece em servidor novo

O driver ODBC Driver 17 for SQL Server precisa estar instalado. Verificar com odbcinst -q -d.

🚫

.env nunca vai pro git

O .env está no .gitignore. Se acidentalmente commitar com credenciais, trocar todas as senhas imediatamente.

⚠️

Supabase CHECK constraint

A coluna frequencia tem constraint que limita os valores aceitos. Se criar nova frequência no código sem atualizar o constraint no Supabase, vai dar erro 400 silencioso ao tentar criar agendamento.

Como Criar um Ambiente de Teste

Opção recomendada: instância separada

  1. WhatsApp de teste: criar nova instância no UAZAPI com número diferente
  2. Banco de dados: apontar para banco de homologação do Protheus (ou banco de desenvolvimento)
  3. Redis: usar REDIS_DB=1 (ou outro número diferente do de produção)
  4. Supabase: criar projeto separado, ou usar schema diferente
  5. OpenAI: mesma chave ou chave separada

.env do ambiente de teste

ENV=development
DEBUG=true
LOG_LEVEL=DEBUG

# Mesmo SQL Server, banco diferente
SQL_SERVER_DATABASE=ProtheusHML

# Mesmo Redis, banco diferente
REDIS_DB=1

# Nova instância UAZAPI
EVOLUTION_INSTANCE_NAME=ComexIM_Teste

Subir na porta diferente

uvicorn main:app --host 0.0.0.0 --port 8001 --reload

Configurar o webhook da instância UAZAPI de teste para apontar para http://seu_servidor:8001/webhook.

ℹ️

O ambiente de teste requer um número de WhatsApp diferente. Não é possível usar o mesmo número de produção — o UAZAPI vincula cada instância a um número específico.

Onde olhar quando algo dá errado

# 1. Ver o erro mais recente
journalctl -u agente-comexim -n 50 --no-pager | grep -i error

# 2. Acompanhar em tempo real
journalctl -u agente-comexim -f

# 3. Ver logs de uma mensagem específica
#    Servidor em UTC — horário de Brasília = UTC-3
#    Mensagem chegou às 14:30 SP = 17:30 UTC
journalctl -u agente-comexim \
  --since "2026-05-29 17:25" \
  --until "2026-05-29 17:35" \
  --no-pager
ℹ️

Atenção ao fuso horário nos logs: o servidor registra em UTC. Brasília é UTC-3. Se a mensagem chegou às 14:30 no Brasil, procure nos logs por 17:30 UTC.
