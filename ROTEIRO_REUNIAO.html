2
O que o sistema faz — visão geral
"Em resumo: o sistema recebe mensagens de WhatsApp, entende o que o usuário quer usando IA, consulta o banco do Protheus, e responde em linguagem natural. O usuário pergunta 'quanto foi vendido em janeiro?' e recebe a resposta formatada, sem precisar abrir nenhum sistema."
"O fluxo completo é assim: mensagem chega pelo WhatsApp → passa por uma camada de autenticação que valida no Protheus se aquele número tem acesso → vai para o agente de IA → o agente decide qual consulta fazer no banco → retorna a resposta pro usuário."
💡 Aqui é um ótimo momento para mostrar ao vivo — manda uma mensagem no WhatsApp e mostra a resposta chegando em tempo real. Vale mais que qualquer explicação.
3
Como foi construído — stack técnico
"A estrutura é: FastAPI como API web — é assíncrono, ideal para lidar com várias mensagens ao mesmo tempo. LangGraph para o agente de IA, usando o padrão ReAct onde a IA raciocina antes de agir. GPT-4o da OpenAI como modelo de linguagem. Redis para cache e memória das conversas. Supabase para agendamentos automáticos. E o banco do Protheus via ODBC para as consultas."
"Uma decisão importante: o sistema é 100% read-only no Protheus. Tudo é SELECT * FROM stored_function(). A IA nunca escreve, nunca altera nada no ERP. Isso foi intencional por segurança."
💡 Essa parte do read-only sempre impressiona e gera confiança. Deixa claro que não tem risco de a IA estragar dados.
4
Como adicionar novas funcionalidades — o mais importante para eles
"Quando vocês quiserem adicionar uma nova consulta — por exemplo, um relatório de RH — o processo é simples em 4 passos:"

Passo a passo para nova funcionalidade

1
Criar a stored function no SQL Server com os dados que quer retornar
2
Registrar a função em app/core/database.py no dicionário FUNCTION_PARAMETERS
3
Criar a tool em app/agents/sql_tools.py — basicamente uma função Python com uma descrição em linguagem natural dizendo quando a IA deve usá-la
4
Adicionar na lista get_tools() no mesmo arquivo
"O ponto mais importante: a IA aprende a usar a nova ferramenta pela descrição que vocês escrevem. Quanto mais clara e específica for a descrição, melhor a IA vai saber quando usar. Não tem treinamento, não tem nada especial — é só texto."
💡 Se tiver tempo, mostra o arquivo sql_tools.py e aponta uma tool existente. Eles vão entender na hora.
5
Como subir em produção — deploy
"O servidor roda Linux com o serviço gerenciado pelo systemd. Para fazer um deploy, o processo é:" 
# 1. Entra na pasta do projeto no servidor
cd /opt/agente-comexim-whatsapp

# 2. Atualiza o código — sempre assim, não só "git pull"
git fetch origin
git reset --hard origin/main

# 3. Reinicia o serviço
systemctl restart agente-comexim

# 4. Confirma que subiu sem erro
journalctl -u agente-comexim -n 20 --no-pager
"Tem um detalhe importante: não usar só git pull para deploy. Em algumas situações ele diz 'Already up to date' mesmo sem ter atualizado os arquivos. O git reset --hard origin/main garante que o servidor fica idêntico ao repositório."
💡 Esse é um dos aprendizados práticos do projeto. Falar isso mostra que você já passou por isso e resolveu.
6
Configuração e variáveis de ambiente — o .env
"Toda a configuração do sistema fica no arquivo .env na raiz do projeto. Ele não vai pro GitHub por segurança — existe um .env.example com todas as variáveis listadas para servir de guia."
"As principais que vocês vão precisar configurar: credenciais do SQL Server, chave da OpenAI, token do UAZAPI, Redis e Supabase. Tem tudo documentado no manual com explicação de cada variável."
7
Cuidados importantes — o que não fazer
"Algumas coisas que aprendi na prática que vale mencionar:"
"Primeiro: não aumentar o REDIS_MEMORY_WINDOW para além de 15. Esse parâmetro controla quantas mensagens ficam no contexto da IA — se ficar muito grande, estoura o limite de tokens da OpenAI quando vários usuários usam ao mesmo tempo."
"Segundo: se mudar de servidor, o primeiro passo é instalar o ODBC Driver do SQL Server. É fácil de esquecer e o sistema não sobe sem ele."
"Terceiro: o Supabase tem um CHECK constraint na coluna de frequência dos agendamentos. Se quiserem adicionar um novo tipo de agendamento no código, precisam atualizar esse constraint no Supabase também — senão vai dar erro silencioso."
"Quarto: se a API da OpenAI retornar erro insufficient_quota, é crédito zerado — não é bug de código. Precisa adicionar crédito em platform.openai.com. O sistema tem retry automático para rate limit normal, mas não tem fallback para crédito zerado."
8
Ambiente de teste — como criar
"Para ter um ambiente de teste separado do de produção, precisam basicamente de três coisas: um número de WhatsApp diferente para criar nova instância no UAZAPI, apontar para o banco de homologação do Protheus, e usar REDIS_DB=1 no lugar do 0 para não misturar os dados."
"O sistema sobe numa porta diferente — por exemplo porta 8001 — e o webhook do UAZAPI de teste aponta pra essa porta. Produção continua na porta 8000 sem interferência."
9
Como ver os logs quando algo dá errado — monitoramento
"Para investigar um problema, os logs ficam no journalctl. Os comandos mais úteis:" 
# Ver erros recentes
journalctl -u agente-comexim -n 50 --no-pager | grep -i error

# Acompanhar ao vivo
journalctl -u agente-comexim -f
"Um detalhe: o servidor registra em UTC, mas o horário de Brasília é UTC-3. Se uma mensagem chegou às 14h aqui, nos logs vai aparecer como 17h."
